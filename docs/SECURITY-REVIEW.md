# 安全与数据安全分析（Microsoft-Email-Manager-Desktop）

> 评审范围：`src-tauri`（Rust 后端：crypto / db / state / commands / accounts_auth / imap / graph / background / export）与 `src`（Svelte 前端）。
> 评审时间：2026-06（对应 git `main`）。结论：**整体加密与存储设计稳健**，无高危漏洞；存在若干中/低风险点与本次新需求引入的权衡，详见下文。

---

## 1. 威胁模型

- **主要保护目标**：本地落盘的邮箱凭据（`refresh_token` / `client_id`）。这些等价于「长期访问该邮箱的钥匙」。
- **主要对手**：拿到 `mem-desktop.db` 文件的人（同机其他用户、备份泄露、丢失设备）。
- **不在范围内**：Microsoft 侧账号安全、用户主密码强度、内核级键盘记录 / 内存取证（有物理+root 权限的对手几乎无法防御）。

---

## 2. 做得好的地方（无需改动）

| 项 | 评价 |
|---|---|
| 密钥分层 | 随机 **DEK** 加密数据，主密码经 **Argon2id** 派生 **KEK** 再包装 DEK。改密码只需重包装 DEK，无需重加密全库——设计正确。 |
| 字段加密 | 每个敏感字段独立 **AES-256-GCM**，`nonce(12)‖ct‖tag`，每次随机 nonce。GCM 认证失败即判定密码错误，避免 padding-oracle 类问题。 |
| 密钥清零 | `DerivedKey` 实现 `Drop` 时 `zeroize`；锁定 / 退出即清空 vault。 |
| Argon2 参数 | 64 MiB / 3 轮 / 并行 1——桌面端合理强度，能显著拖慢离线爆破。 |
| SQL | 全部走 `rusqlite` 参数化（`?1..`）；表名来自内部固定集合并在 `add/delete_option` 校验，无注入面。 |
| TLS | IMAP 用 `native-tls` 默认构造、Graph/token 用 `reqwest` rustls，**均默认校验证书**，无 `danger_accept_invalid_certs`。 |
| 邮件正文沙箱 | 详情用 `<iframe sandbox="allow-same-origin">`（**未开 `allow-scripts`**），邮件内 JS 不执行；`block_remote_images` 时向邮件 HTML 注入 `img-src data: cid:` 拦截追踪像素。 |
| 外链打开 | 新增 `open_url` 仅放行 http/https 且过滤 shell 元字符，用 `rundll32 FileProtocolHandler`，无命令注入。 |
| 导出 | 含凭据导出有明文告警，且（本轮起）需二次身份校验。 |

## 3. 发现的问题与建议

### 🟠 中风险

1. **`standalone_2fa`（独立 2FA）模式数据保护偏弱**
   - 该模式下数据由「设备密钥」兜底加密，安全性弱于「密码+2FA」。引导页已有警告。
   - 建议：保留警告即可；可在「关于/安全」再次提示该模式不抗「拿到设备文件」的对手。

2. **明文 `refresh_token` 导出**
   - 即便有告警，明文导出文件一旦泄露等于交出邮箱长期访问权。
   - 建议：默认强制加密导出（明文需再勾一次「我已知风险」）；本轮已加「导出前二次校验」，方向正确。

3. **（本次新需求）锁定后仍后台收信 → 削弱「锁定」语义**
   - 要在锁定后继续拉新邮件，后端必须在锁后仍保留可用的凭据/令牌。这与「锁定即清空密钥」直接冲突。
   - 缓解方案见第 4 节（建议只保留**短期 access_token**，不保留 refresh_token）。

### 🟡 低风险

4. **无登录尝试限速 / 锁定**：`unlock` 可无限重试。离线爆破已被 Argon2 拖慢，应用内重试也需先有 DB 文件，风险有限。建议：可加「N 次失败后指数退避」提升体感安全。
5. **TOTP 可在时间窗内重放**：`totp_verify` 未记录「已用过的码」。建议：缓存最近成功的 `(counter)` 拒绝重复（防同一码在 30s 窗内重复使用）。
6. **剪贴板残留**：揭示的凭据 / 验证码复制后长期留在系统剪贴板，锁定后依然可被读取。建议：复制敏感内容后 N 秒自动清空剪贴板。
7. **主密码明文 `String` 不清零**：IPC 反序列化得到的 `password: String` 用后未 `zeroize`（DEK 已清零）。属深度防御项，收益有限。
8. **应用级 CSP `img-src` 含 `https:`**：这是为「关闭远程图片拦截时邮件 iframe 能加载远程图''”而保留，主界面本身不加载远程图。**属预期行为**，保持现状即可（若收紧会破坏该开关）。

### ℹ️ 运维 / 可用性（非漏洞但需知会）

9. **5 秒默认轮询过于激进**：对每个开启的邮箱每 5s 拉一次 IMAP/Graph，可能触发 Microsoft 限流甚至风控（异常高频）。已按你要求设为默认 5s，但建议：UI 上提示「过低间隔可能被限流」，并对失败做退避。

---

## 4. 针对「锁定后仍提示」的安全设计建议

要兼顾「锁后仍能提醒新邮件」与「锁定的意义」，推荐**令牌即焚**方案：

- 锁定时**只**把被监视邮箱的**短期 `access_token`（约 1 小时有效）**留存在一块独立内存里（不写盘）；**绝不保留 `refresh_token`**。
- 锁后后台循环用这些 token 拉信；**token 过期即无法续期**（refresh_token 在锁里拿不到），该邮箱自动停更——天然限制暴露时长（≤1h）。
- 系统通知**只显示邮箱地址 + 发件人**，不含主题/正文/验证码（锁态下不解密内容）。
- 解锁后清空该内存、恢复正常。

> 与「保留 refresh_token 长期收信」相比，本方案把锁后暴露面从「永久长钥」降到「≤1 小时短钥、且无主题内容」，是更可接受的权衡。

---

## 5. 建议优先级

1. （已做）修复密码输入「未响应」——把 Argon2 移出主线程。
2. （本轮新增）锁后通知采用第 4 节「令牌即焚 + 仅显示发件人」。
3. 剪贴板自动清空（验证码/凭据）。
4. TOTP 防重放；失败退避。
5. 默认导出强制加密。
